# Halal Damu Bot - Техникалық Архитектурасы

## 1. Технологиялық стек (Tech Stack)
- **Frontend (Клиенттік бөлік):** React 18, TypeScript, Tailwind CSS, Vite.
  - *Пайдаланушы интерфейсі:* `shadcn/ui` компоненттері.
  - *Маршрутизатор:* `react-router-dom`.
- **Backend (Серверлік бөлік):** Node.js, Express.js.
  - *Логика:* API эндпоинттері (`/api/admin/...`), Telegram (бұдан былайғы байланыстар).
  - *Жасанды Интеллект (AI) және Ембеддингтер:* Ресми **Google Gen AI SDK (`@google/genai`)** Vertex AI режимінде қолданылады. Барлық Google AI Studio сұраулары немесе API кілттеріне бағынышты жүйе толығымен жойылды, орныне GCP IAM рұқсаттары немесе Service Account арқылы тікелей Vertex AI платформасында жұмыс істейтін жаңа бірыңғай СДК енгізілді.
- **Деректер қоры (Database):** Google Cloud Firestore (NoSQL).
- **Хостинг / Инфрақұрылым:** Google Cloud Run (Docker контейнері арқылы). Төлемдер мен баланстар тікелей "momyn-t1" GCP жобасына байланған корпоративтік Billing шотынан (IAM рұқсаттары немесе Service Account арқылы) автоматты түрде алынады.

## 4. Деректер алмасу логикасы (Data Flow)
1. **Frontend -> Backend:**
   CMS-те деректер (мысалы, "Сақтау" батырмасы) өзгертілген кезде `PUT /api/admin/search_companies/:id` немесе `PUT /api/admin/search_ingredients/:id` бағыты бойынша JSON payload жіберіледі.
2. **Backend -> Firestore:**
   Express сервері деректерді қабылдап, валидация жасайды (мысалы, `tags` немесе `aliases` жиым екенін тексереді) және `docRef.update()` арқылы базаға жазады.
3. **Backend -> Frontend:**
   Сервер `200 OK` мен жаңартылған нысанды қайтарады. Клиент `AdminCMS` компонентінде күйді (state) жаңартып, өзгерісті экранда тікелей көрсетеді.
4. **Мекемелерді сыртқы API-ден синхрондау (Companies Sync Engine):**
   - **Автоматты түрде өзгеріссіз синхрондау:** Күнделікті сағат 03:00-де (Азия/Алматы уақытымен) Express серверінде `node-cron` арқылы инкрементті жаңарту (`runSync`) іске қосылады.
   - **Мультимодальді және Ақылды Инкрементті Жаңарту (Multimodal & Incremental Update):** Скрипт `https://halaldamu.kz/wp-json/halal-bot/v1/companies?lang=kz` сілтемесінен мекемелерді тартады және әр мекеменің `updated_at` белгісін Firestore-да бар деректермен салыстырады. Жаңа немесе өзгерген/векторы жоқ мекемелерлі толықтай `gemini-embedding-2-preview` арқылы векторлаймыз. Ембеддинг жасау барысында мекеме мәтінімен қатар оның басты суреті (`featured_image` немесе `logo_image`) Node.js серверінде автоматты түрде жүктеліп, Base64 ретінде **multimodal input** ретінде жіберіледі.
   - **Параллельді Жеделдету (Concurrency Batching - 15X Faster):** Синхрондау процесін максималды жылдамдату мақсатында мекемелер 15-тен топталып (chunk size: 15), `Promise.all` арқылы параллельді түрде векторланады. Сұраныстар арасындағы Google Vertex AI API квотасын қорғау үшін әр топтың арасында 1 секундтық кідіріс жасалады.
   - **Қолмен іске қосу (Manual Trigger - HTTP Streaming):** Әкімшілік мақсатта `GET /api/admin/sync-now` сілтемесі арқылы синхрондау сұранысын жіберуге болады. Контейнердің ұйықтап қалуын (freeze) болдырмау және тайм-ауттардан сақтау үшін бұл роут Express-тің HTTP Streaming (`Transfer-Encoding: chunked`) технологиясына негізделген. Ол синхрондау барысын нақты уақытта браузерге ағынмен (chunk) жазып жіберіп отырады. Егер сұранысқа `?force=true` параметрі берілсе, барлық мекемелерге жаңаша векторлау жүргізіліп, Firestore-ға күштеп жазылады. Сондай-ақ вектор базада `FieldValue.vector` арқылы VectorValue типімен сақталады. Сұраулар мен іздеулерде (`searchData` және `test-search`) іздеу векторы міндетті түрде дәл осы форматқа сәйкестендіріліп жіберіледі.
   - **Сервер қосылғанда іске қосылу (Startup Trigger):** Әзірлеу немесе тестілеу режимінде (`development` немесе non-production) сервер алғаш іске қосылған кезде бұл синхрондау утилиті автоматты түрде бір рет орындалады, бұл локальді деректердің өзектілігін қамтамасыз етеді.
   - **Кэш баптауы (In-Memory Cache):** Синхрондау сәтті аяқталған соң барып, Іздеу жады (`loadCache(true)`) жаңартылып, боттың жауап қату жылдамдығы барынша жоғары (sub-millisecond) деңгейде сақталады.
   - **Синхрондау мониторингі мен диагностикасы (Sync Status API):** Нақты уақыт режимінде деректердің өзектілігін бақылау мақсатында `GET /api/admin/sync-status` сілтемесі қосылды. Бұл роут Firestore-дағы мекемелердің жалпы санын (`total_companies`), векторланған құжаттардың нақты санын (`embedded_companies`), соңғы синхрондау барысында шыққан жүйелік қателікті (`last_sync_error`) және деректер типі мен құрылымын растау үшін кез келген 5 мекеме үлгісін («has_embedding» бульдік өрісімен бірге) JSON есебі ретінде қайтарады.

## 5. Суретті тану архитектурасы және AI Клиент (Image Recognition & Fast OCR/Brand Extraction Architecture)

Жүйе сурет арқылы өнімді, брендті немесе тағамдық құрамды талдап, дерекқордан лезде тауып беру үшін **gemini-1.5-flash немесе оған сәйкес жылдам модельдердің тездетілген бір қадамдық JSON (Single-turn JSON output) режімін** және орталықтандырылған AI Клиентті қолданады:
- **Толықтай Google Gen AI SDK (`@google/genai`):** Төлем баланстары мен 429 RESOURCE_EXHAUSTED қателерін толықтай шешу мақсатында жүйе толықтай Google Gen AI кәсіби SDK-інің Vertex AI режиміне көшірілді. Барлық мәтіндік генерация, стриминг және эмбеддингтер ("gemini-embedding-2-preview") "momyn-t1" жобасы мен "us-central1" аймағына негезделіп, орнатылған Service Account арқылы жүзеге асырылады. Ескі AI Studio және оның API кілттері (GEMINI_API_KEY) қолданылмайды.
- **Орталықтандырылған Сурет Талдау (Consolidated Image Search):** Сурет талдау жүйесі (`agenticImageSearch.ts`) де осы жаңа басты `ai` Vertex AI-ға бейімделген клиентінен шақырылып, Vertex AI басымдылықтарына толығымен бағынады.
- **Сенімді және Жетілдірілген Әрекет Көрсету (Robust Native Chat Action with sendChatAction):** Пайдаланушыларға жауапты дайындау барысында фантомдық қателіктер мен 90,000ms тайм-ауттарды болдырмау үшін Telegram-ның стандартты `sendChatAction("typing")` утилиті қолданылады. Бұл ресми Telegram Bot API серверлерінде қателіктерсіз әрі өте жылдам жұмыс істеуге кепілдік береді. Бизнес-қосылымдар мен топтық чаттарда кез келген байланыс мәселесін жойып, боттың жауап жылдамдығын максималды арттырады.
- **Тақырыптарды ақылды өзгерту (Daraq-Style Smart Renamer):** Топтар мен тақырыптарда тақырып атауын автоматты түрде өзгерту логикасы **Daraq** архитектурасына сәйкес жетілдірілді. Егер пайдаланушымен жәй сөйлесу (chitchat/conversational) орын алса, топ шуламауы және артық өзгерістер болмауы үшін тақырып атауы өзгеріссіз қалады. Ал егер нақты іздеу (knowledge/product search), маңызды сұраныс, суретті AI арқылы талдау немесе геолокациялық (орын) сұраныс жіберілсе, жүйе оны Gemini LLM көмегімен қысқа әрі нұсқа (эмодзи қоса отырып, мысалы сурет үшін сәйкес анықтама мен орын үшін "📍 Айналадағы жақын орындар/шешімдер") ретінде автоматты түрде өзгертіп отырады.
- **Автоматты HTML тегтерін жабу тетігі (Auto HTML Tag Closer):** Стриминг кезінде Telegram-ның "mismatched entities" (ашық қалған тегтер) қателіктерін болдырмау үшін әр хабарлама жіберілу алдынде барлық ашық қалған HTML тегтері (`<b>`, `<i>`, `<code>`) автоматты түрде өзінің жабылу таңбаларымен жабылады. Бұл стримингтің үзіліссіз, біртіндеп, әдемі шығуын қамтамасыз етеді.
- **Маркдауннан тазарту (Robust MD-to-HTML Sanitation):** Жауап барысында пайда болатын `**`, `##`, `__` секілді маркдаун белгілері мен артық жұлдызшалар автоматты түрде тазартылып, таза HTML форматында беріледі.

## 6. Ерекше мүмкіндіктер (Key Features)

- **Икемді маршрутизатор (Intelligent Router & Classifier):** Бұл маршрутизатор **Daraq** architecture-сындағы репо логикасымен одан әрі жетілдірілді. Код ішіндегі барлық қолдан тізілген кілт сөздер сүзгілері толығымен жойылып, адамдардан келген кез келген хабарлама КҮМӘНСІЗ бірінші болып Gemini-дің интеллектуалды жіктеуіш моделіне бағытталады. Модель хабарламаны оқып БІРДЕН жауап ретінде әрекетті (`chat` немесе `search`) және сәйкес шешімді JSON форматында қайтарады.
  - **Жақсартылған ниет бағыттау (Intent Routing Rules):** Пайдаланушы "ұсыншы", "қайда бар", "жегім келеді", "орын бар ма", "ішетін жер", "кафе", "ресторан" сияқты тамақтану орындарын немесе ұсыныстар сұраған мағынадағы хабарламалар жіберсе, бұл КҮМӘНСІЗ `search` (іздеу) ниеті болып бағытталады және сұраққа сәйкес тазартылған кілт сөз ("Ресторан Алматы", т.б.) `query` өрісіне шығарылады.
  - **Дайын жауаптар:** Егер модель сұранысты жай сөйлесу (`chat`) деп таныса, ол екінші рет LLM шақыруын күтпей, бірден дайын жауап мәтінін қоса дайындап шығарады. Ал іздеуде (`search`) барлық артық сөздер алынып тасталып, бренд/мекеме атауы Firestore базасынан тікелей векторлық әдіспен ізделеді. Бұл кездейсоқ қате сәйкестіктердің алдын алады.
- **Мекеме мен Қоспаны ажырату және Е-кодтарды қатаң іздеу логикасы:**
  - **Базаларды толық ажырату (Search Isolation):** Егер сұраныс жиілікке/тамақтануға қатысты кілт сөздерден ("орын", "кафе", "дәмхана", "ресторан", "жегім келеді", "пицца", "кофе", "бургер" т.б.) немесе қала аттарынан тұрса, бұл сұраныс ингредиенттер (Е-кодтар) дерекқорынан іздеуді мүлдем өткізіп жібереді де, тек қана "search_companies" мекемелері базасынан векторлық ізделеді.
  - **Қатаңдатылған Е-код және Ингредиент логикасы (Strict Ingredient Matching):** "Кофе" деп іздегенде "токоферол" сияқты химиялық қоспалармен substring арқылы шатастырмас үшін жаңа сүзгі утилиттері енгізілді. Ингредиенттерден тек нақты E-код үлгісі (мысалы, "Е471") немесе химиялық термин жұрнақтары ("ид", "ат", "ит", "ол", "ин", "за", "оза", "ан") мен белгілі химиялық су/желатин сияқты атаулар болса ғана ізделеді. Бұл жалпы тамақтану терминдерімен ингредиент сөздерінің кездейсоқ соқтығысуын болдырмайды.
- **Еркін сөйлесу режімі (Chat Mode with Memory):** Firestore дерекқорында сақталған пайдаланушы чат тарихын (`chat_history`) ескере отырып, Gemini 3.5 өте мәдениетті әрі сыпайы (Сіз деп) халал тұрғысынан ақыл-кеңес береді. Сымбат атты ерекше пайдаланушы үшін арнайы махаббат пен еркелетуге негізделген Easter-Egg режімі құрылған.
- **Кері байланыс жүйесі (Feedback System):** Бұрын қолданылған "👍 Пайдалы" және "👎 Қате" кері байланыс батырмалары чат интерфейсін таза әрі жинақы сақтау үшін пайдаланушылардың сұрауы бойынша толықтай жойылды.
- **Дерекқор іздеу сәйкестігі мен батырмалар:**
  - Бірнешеу табылса, оларды интерактивті пейджинация тізімі ретінде таңдауға ұсынады.
  - Е-кодтар табылса, пайдаланушы бір батырмамен әр қоспаның анықтамалығын дәл аша алады.
- **Орналасқан жер батырмасының тұрақтылығы (Button Persistence):** Пайдаланушы әр түрлі сұрақтар қойғанда немесе AI-мен сөйлескенде де, жеке чатта `📍 Айналадағы халал мекемелер` батырмасы ешқашан жоғалып кетпейді, ол барлық жауап хабарламаларында клавиатура ретінде сақталып отырады.
- **Пейджинация (Pagination):** Браузер немесе чат жүктелмеуі үшін тізімдерді бөліп көрсету.
- **Терминал (Terminal):** Жүйенің ішкі логикасы мен логтарын көрсететін бақылау интерфейсі.
- **Икемді форма:** Форманың элементтері тік бағытта (vertical-stack) бір бағанға жиналған, әртүрлі құрылғыларда интуитивно жұмыс істейді.
- **Топтардағы тақырыптарды (Forum Topics) қолдау:** Тікелей супертоптар ішіндегі белгілі бір тақырыптарда (Forum Threads) бот келген хабарламаларға дәл сол тақырыптың (`message_thread_id`) ішінде жауап қайтарады. Бұл Telegraf-тың барлық хабарлама жіберу әдістерін және `ctx.telegram.callApi` прототипін қауіпсіз тұзаққа салу (hook/intercept) арқылы жүзеге асырылған.
  - *Орналасқан жер батырмасы мен үйлесімділік:* Жеке чаттардағы орналасқан жерді бөлісу клавиатурасын сақтау және Telegram Desktop секілді сыртқы клиенттердің текстік ерекшеліктерімен үйлесімділік үшін `📍 Айналадағы халал мекемелер` текстік командасын тікелей өңдеп гео-интерактивті бағыттау мүмкіндігі қорғалған. Пайдаланушыларға ыңғайлы болу үшін негізгі клавиатура қос қатарды (Double-row structure) иеленді: 1-ші қатарда ұялы телефондар үшін GPS-ті бірден жіберетін `📍 Менің орнымды жіберу` (location button request), ал 2-ші қатарда Desktop/нұсқаулыққа арналған `📍 Айналадағы халал мекемелер` мәтіндік батырмасы тұрақты орналастырылды.
  - *Топтық чаттар мен арналардағы автоматты сүзу (Group Mention Filter):* Топтарда боттың басқа пайдаланушылардың өзара әңгімесіне кедергі жасамауы үшін `groupMentionFilterMiddleware` енгізілді. Бот топтық чаттарда  тек оны `@username` ретінде тікелей атап өткенде немесе боттың өз хабарламасына Reply (жауап) арқылы жазғанда ғана сұранысқа құлақ асады, әйтпесе хабарламаны елемейді.
  - *Іздеу мәтінін тазарту (Mention Cleaning):* Пайдаланушы топтық чатта бот атауын көрсеткенде (мысалы, `@HalalDamu_bot Шұжық`), сұранысты өңдеу алдында `@username` тіркесі мәтіннен автоматты түрде алынып тасталады. Бот өңдегішінің таза іздеу кілттерімен жұмыс істеуіне кепілдік береді.
  - *Топтарда автоматты тақырып құруды өшіру:* Топтық чаттарды ретсіз тақырыптармен толтырмау және топ ішіндегі ыңғайлылық үшін жаңа сұрақтарға автоматты түрде жеке Forum Topic (тақырып) құру логикасы өшірілді. Бот кез келген топтық чатта немесе Forum Topic ішінде өз орнында ғана жауап жазады.

## 7. Боттың іске қосылу режимі (Bot Launching Mode)

Қазіргі нұсқада Telegram бот тек **Long Polling (ұзын сұрау)** режимінде іске қосылады. Бұл кез келген ортада (соның ішінде AI Studio-ның жабық локалды ортасы мен Cloud Run өндірістік контейнерінде) сервердің сыртқы URL мекенжайын тіркеуді қажет етпестен, Telegram серверлерінен хабарламалар мен жаңартуларды автоматты түрде тікелей тартып алуды қамтамасыз етеді және желілік бағыттау мәселелерін немесе Webhook арнасын баптау қажеттілігін толығымен сейілтеді.

- **Ерекшеліктері:**
  - **Масштабталуы:** Тестілеу орталары мен локалды тұғырларда ешқандай конфигурациясыз бірден іске қосылады.
  - **Кезек тазарту (Pending updates cleanup):** Бот іске қосылғанда `dropPendingUpdates: true` параметрі қолданылады. Бұл бот өшіп тұрған немесе жаңартылып жатқан аралықта жиналған ескі хабарламалар кезегін тазартады, осылайша бот қайта қосылғанда ескі хабарламаларды қайталап өңдемейді және жүйелік жүктемені азайтады.
  - **Ақауларды автоматты түзету:** Токенде қайталанған идентификаторлар (ID:ID:SECRET) кездескен жағдайда бот оны ішкі алгоритммен түзетіп барып іске қосылады.

## 8. Гибридті және Векторлық іздеу архитектурасы (Hybrid & Pure Vector Search Architecture)

Жүйеге іздеу сапасын барынша жетілдіру үшін ең заманауи семантикалық және классикалық әріптік іздеу әдістерін біріктіретін **Гибридті іздеу (Hybrid Search)** механизмі енгізілді. Қазіргі уақытта векторлық және фаззи іздеу толықтай біріктіріліп, бірге жұмыс істейді:

1. **Сұранысты векторлау:** Пайдаланушы іздеу мәтіні `gemini-embedding-2-preview` моделі арқылы көп өлшемді вектор түріне аударылады (1536 dimension, нөлдермен толтырусыз/zero-padding-сіз нативті).
2. **Семантикалық іздеу (Semantic Vector Search):** Firestore Node.js SDK `findNearest` интерфейсі арқылы `search_fields.embedding` өрісінен Cosine айырмашылығы негізінде ең жақын 10 мекеме анықталады. Бұл сұраныстың нақты жазылуына қарамастан (синонимдер, мағыналық ұқсастықтар) нәтижелерді табуға мүмкіндік береді.
3. **Параллельді Фаззи іздеу (Fuzzy Search):** Базадағы барлық белсенді мекемелер мен кэштелген тегтер/синонимдер арасынан `fuzzball` арқылы әріптік сәйкестік ізделеді. Әріп санының шектеуі 2 әріпке дейін түсірілген (`wClean.length >= 2`), ал ұқсастық шегі қатаң түрде 50-ге дейін төмендетілген. Бұл қысқа сұраныстардың (мысалы, "Сеул" деп іздегенде "Сеул Ким" мекемесі) 100% табылуын қамтамасыз етеді.
4. **Ақылды Біріктіру және Ұпай беру (Smart Merging & Score Ranking):** Векторлық және Фаззи іздеуден келген нәтижелер ID бойынша біріктіріліп, сәйкестік күйіне қарай ортақ ұпайлар үлестіріледі (Score: 100 — қос іздеу де сәтті; 80 — нақты классикалық сәйкестік; 60 — семантикалық векторлық сәйкестік; 40 — фаззи сәйкестік) және басымдық бойынша сұрыпталады (ranking).
5. **Геолокациялық сүзгі мен сұрыптау:** Егер пайдаланушы орналасқан жерін (координаттарын) қоса жіберсе, табылған семантикалық нәтижелер Firestore-дағы `GeoPoint` немесе олардың мекенжайына қарай қашықтық (Distance) бойынша қайта сұрыпталып, ең жақын тұрғандары алдыңғы кезекке орналастырылады.
6. **Үздік Нәтижелерді Шектеу:** Соңғы пайдаланушыға тек ең маңызды, сұрыпталған үздік 5-7 нәтиже ұсынылады.

## 9. Жетілдірілген Агенттік Чат және Құрал Шақыру (Agentic Chat & Function Calling RAG Mode)

Іздеу жүйесін қарапайым құрғақ тізімдерден нағыз адам-кеңесші деңгейіне жеткізу және боттың жауап жылдамдығын өте жылдам ету үшін **Function Calling / Tool Use** негізіндегі бірыңғай Агент режимі енгізілді:
1. **Бірыңғай Чат Сессиясы (Unified Chat Session):** Ескі бөлек Intent Classifier және RAG Generator жүйелері толығымен өшірілді. Пайдаланушы жазған бойда, оған Vertex AI (`@google/genai`) арқылы бірыңғай Чат сессиясы іске қосылады (`aiClient.chats.create`).
2. **Контекст пен Тарихты Толық Есте Сақтау (Firestore History Sync):** Чат сессиясын бастаған кезде Firestore-дан пайдаланушының алдыңғы сөйлесу тарихы (`history` массиві) алынып, Gemini-ге толық беріледі. Бұл сөйлесу контекстін 100% сақтайды.
3. **Іздеу Жүйесі Құрал Ретінде (search_halal_companies Tool):** Чат моделіне аспап (tool) ретінде біздің `searchData` іздеу функциямыз ресми түрде тіркелді:
   - **name:** `search_halal_companies`
   - **description:** Қазақстандағы халал сертификаты бар кафе, ресторандар мен қоспаларды семантикалық іздеу құралы.
   - **parameters:** `query` іздеу сұранысы.
4. **Агенттің Жүйелік Мінезі (System Instructions):** Gemini Агенті ҚМДБ Халал Даму ұйымының өте ақылды, жылы және ресми кеңесшісі сияқты әрекет етеді. Егер пайдаланушы мекеме, дәмхана немесе тағам түрін іздесе, ол бірден құралды шақырып іздейді де, нәтижесін әдемі HTML форматында (<b>, <i>, 🗺 2GIS сілтемелері) мәтін ішіне кірістіріп, әңгімелесе отырып ұсынады.
5. **Орындалу Жылдамдығы мен Оңтайландыру:** Екі бөлек LLM шақыруының орнына бір ғана Чат шақыруы орындалатындықтан, боттың жауап беру жылдамдығы барынша жеделдеп, нағыз тірі кеңесші деңгейін көрсетті. Тілдесу барысында Telegram-дағы HTML entities қауіпсіздігі үшін `Auto HTML Tag Closer` тетігі сақталып қалды.
7. **Терапиялық және диагностикалық құралдар (Verbose Diagnostic Tools):**
   - **Егжей-тегжейлі Консольдік Логтар:** `searchData` функциясы орындалғанда сыртқы сұраныстың шикі түрі (Raw Query), стоп-сөздерді тазартқаннан кездегі күйі (Processed Query), ембеддингтің құрылу қадамы, вектор ұзындығы және Firestore findNearest әрекеті толық бақыланып, консольге жазылады.
   - **Firestore Қателіктерін Толық Көрсету (Error Trace):** Векторлық іздеу барысында Firestore-ға қатысты қателер орын алса (мысалы, қажетті векторлық индекстің болмауы), толық `stack trace` жүйелік логқа шығарылып, түзету сілтемесін (Index creation link) лезде анықтауға жол ашады.
   - **Қадамдық Өздік Диагностика Роуты (Step-by-step Self-Diagnostic API):** Әкімшілік мақсатта `GET /api/admin/test-search` арнайы сілтемесі құрылды. Бұл роут браузерде ашылған кезде асылып қалмай, бірден 200 OK статусымен JSON есеп қайтарады. Ол жүйеның үш негізгі деңгейін (1: Firestore байланысы, 2: Gemini Embedding API жұмыс қабілеті, 3: Векторлық findNearest сұранысы мен индекстің бар-жоғы) жеке-жеке тексеріп, нәтижелерін нақты әрі қауіпсіз көрсетеді.
